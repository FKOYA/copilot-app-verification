# GitHub Copilot アプリ検証計画

## 1. 目的

GitHub Copilot アプリ(テクニカルプレビュー)について、以下を明らかにする。

- 何ができるのか(機能の実態把握)
- 他ツール(VSCode の Copilot Chat、GitHub.com の Copilot、Claude Code 等)との違いは何か
- 実際の開発でどう活かせるか(導入判断の材料)

検証手順・結果を再現可能な形で記録し、後から参照・共有できる資産として管理する。

## 2. 背景(前提知識の整理)

- GitHub Copilot アプリは、並列ワークストリーム・GitHub 統合・PR ライフサイクル管理を 1 か所にまとめた、エージェント駆動型開発用のデスクトップアプリ(macOS / Linux / Windows 対応)
- Copilot CLI 上に構築され、GitHub とネイティブ統合されている(リポジトリ・ブランチ・CI が追加セットアップなしで動く)
- 主な機能: 並列ワークスペース、セッションモード(対話型 / プラン / 自動操縦)、Issue・PR・CI のアプリ内管理、モデル選択、MCP・スキル等のカスタマイズ、スケジュールされたワークフロー、クイックチャット
- **テクニカルプレビューであり変更される可能性がある** → 計画を作り込みすぎず、検証しながら構造を育てる方針を取る
- 利用条件: Copilot Business / Enterprise は利用可、Pro / Pro+ は待機リスト申請が必要

差別化の仮説(※検証前の仮説。結論は検証後に事実ベースで書き直す):

1. GitHub ネイティブ統合の深さ(Issue / PR / CI 起点でワークフローがアプリ内完結)
2. 並列エージェント実行
3. IDE 非依存のエージェント駆動開発(「書く」より「指示する」)

## 3. リポジトリ設計

### 基本方針

**検証専用リポジトリ 1 つで、作業(ブランチ・PR)と結果(ドキュメント)を一体運用する。**

理由: Copilot アプリの検証価値は「Issue → PR → マージという GitHub フローに実際に乗せて何が起きるか」を見ることにある。作業と結果を別リポジトリに分けると、「この PR がどう作られたか」という証跡と記録の紐付けが切れて参照性が落ちる。

### 構成

```
copilot-app-verification/
├── README.md            # 目的・方針・全体計画・このリポジトリの読み方(更新頻度: 低)
├── LOG.md               # 検証結果の時系列生ログ(検証のたびに追記)
├── docs/                # ログが溜まってから再構成する最終アウトプット(最初は空)
├── sandbox/             # エージェントに実際に触らせる検証用コード
└── .github/
    └── ISSUE_TEMPLATE/
        └── verification-task.md   # 検証タスクの Issue テンプレート
```

### 設計上の注意点

- **最初からカテゴリ別ドキュメントに分割しない。** 最初は LOG.md 1 本で運用し、パターンが見えてきた段階で docs/ に切り出す。構造は検証の"結果"として後から生まれるもの
- 検証で作ったブランチ・PR は**マージせずクローズして証跡として残す**。溜まったら定期的に整理
- **例外: 実プロジェクトでの検証。** 既存の本物のコードベースで試す場合は実プロジェクト側で実施し、結果だけを検証リポジトリの docs/ に転記する(実プロジェクトの履歴を汚さない)。サンドボックスだけでは大きな既存コードベース・実運用 CI・組織ポリシー下での挙動が検証できないため、この枠は必ず設ける

## 4. どこに何を書くか(役割分担)

| 場所 | 書く内容 | 更新頻度 |
|---|---|---|
| **README.md** | 目的、背景・スコープ、進め方の方針(Step 0〜5)、リポジトリの読み方・運用ルール、検証項目と優先度の一覧表 | 低(計画・ルールの置き場) |
| **Issues** | 「今何をすべきか」のタスク管理。検証タスクを 1 件ずつ Issue 化し、クローズで進捗を可視化 | 高(日々動く) |
| **LOG.md** | 実際にやった検証の記録。1 検証 = 「検証手順 → 期待結果 → 実際の結果 → 気づき(特に異常系)」の 4 段構成で追記 | 検証のたび |
| **docs/** | LOG が溜まってから再構成する機能別・観点別の整理、他ツール比較、実開発での活用まとめ | 後半に集中 |
| **sandbox/** | エージェントに触らせる検証用コード | 検証内容次第 |

Issue をタスク管理に使う理由: Copilot アプリ自体が Issue 起点でエージェントにタスクを渡す設計のため、「検証タスクを Issue 化 → アプリの Inbox から拾わせて実行 → 結果を LOG.md に記録 → Issue クローズ」という流れで**検証対象とタスク管理ツールを兼ねられる**。

## 5. 検証項目と優先度

| 優先度 | 項目 | 検証内容 |
|---|---|---|
| **高** | GitHub 統合 | Inbox 起点のタスク実行、PR 作成〜レビュー〜マージの一連の流れ、CI チェック結果の表示 |
| **高** | 並列セッション | 複数セッション同時実行時の速度・リソース消費、ブランチ競合の有無、UI の切り替えやすさ |
| **高** | スケジュールワークフロー | 定型タスク(Issue 自動整理、リリースノート生成など)の設定と実行の安定性 |
| 中 | セッションモード | 対話型 / プラン / 自動操縦それぞれの挙動差、承認フローの違い |
| 中 | カスタマイズ | MCP サーバー接続、スキル追加、拡張機能・プラグインの動作 |
| 中 | モデル選択 | 使えるモデルの種類、モデルごとの精度・速度・コスト感 |
| 低 | 基本セットアップ | 既存記事で手順化済みのため深追いしない(自環境での差分のみ記録) |
| 低 | クイックチャット | セッションなしの気軽な質問利用、コンテキスト有無による回答品質差 |
| 低 | セキュリティ / ガバナンス | パブリックコード一致ポリシーの挙動、プランによるアクセス制御(必要になったら) |

優先度の考え方: **他ツールにない差別化要素 = 検証価値が高い**ものを「高」に置く。既存記事で把握できるものは低優先。

### 異常系を必ずセットで検証する

各項目は「うまくいくケース」だけでなく、以下の**失敗モード**を必ず併せて記録する。実運用での導入判断に最も効くのはここ。

- エージェントが誤ったコードを書いた場合の取り消し方・戻し方
- 自動操縦モードで想定外の変更をした場合の被害範囲
- 権限周りの誤爆(意図しないリポジトリ・ブランチへの操作)
- セッションの中断・失敗時の状態

## 6. 進め方(Step 0〜5)

1. **Step 0: アクセス可否の確認** — 現在のプラン(Business / Enterprise / Pro)でアプリを使えるか確認する。Pro / Pro+ の場合は待機リスト申請。**アクセス確定まで作り込みは最小限に留める**
2. **Step 1: 最小構成でリポジトリ作成** — README + LOG.md + Issue テンプレートのみ。docs/ の中身は作らない
3. **Step 2: 高優先項目の検証** — GitHub 統合・並列セッション・ワークフローを、正常系 + 異常系セットで検証し LOG.md に記録。検証タスクは Issue 化してアプリの Inbox 経由で実行
4. **Step 3: docs/ への再構成** — LOG にパターンが見えてきたら、機能別・観点別ファイルに切り出す
5. **Step 4: 実プロジェクトでの限定検証** — 実際の(または近い規模の)既存プロジェクトで試し、結果を docs/ に転記。サンドボックスとの挙動差を記録
6. **Step 5: 結論の執筆** — LOG / docs の事実に基づいて「他ツールとの差別化」「実開発での活用方法」をまとめる。検証前の仮説(2 章)と突き合わせて答え合わせをする

## 7. LOG.md の記録テンプレート

```markdown
## YYYY-MM-DD: <検証項目名> (Issue #nn)

### 検証手順
1. ...

### 期待結果
- ...

### 実際の結果
- ...

### 気づき(異常系・制約・意外な挙動)
- ...
```

## 8. 運用ルールまとめ

- 構造は先に作らず、検証結果に応じて育てる
- 検証は Issue 起点で行い、Copilot アプリ自体のワークフローに乗せる
- ブランチ・PR は証跡としてクローズ状態で残す
- 異常系のない検証記録は完了扱いにしない
- 「実開発での活用」「他ツール比較」の結論は、検証後に事実ベースで書く(仮説先行のバイアスを避ける)
- テクニカルプレビューのため、アプリの仕様変更があれば LOG.md に日付付きで記録し、影響する docs/ を更新する
